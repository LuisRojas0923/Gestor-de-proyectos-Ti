## 4. Hallazgos bloqueantes

### P0-1 — Login emite JTI, pero no persiste el mismo JTI de forma atómica

`ServicioAuth.crear_token_acceso()` añade un JTI al JWT (`servicio.py:142-185`), pero `login_router.py:210-223` llama a `registrar_sesion()` sin pasar ese JTI. La fila de `sesiones` queda con `jti=NULL` para el login web.

Además, el `sesion_service.py` actual ignora la sesión `db` del request y crea otra `AsyncSessionLocal` (`:37-62`), captura cualquier excepción y la convierte en un warning (`:63-67`). Por tanto, el login puede devolver un JWT que no quedó registrado, mientras una futura validación de WS basada en JTI lo rechaza o no puede revocarlo.

**Gate:** el servicio de emisión debe generar un único JTI, usarlo en JWT y DB, persistirlo en la misma transacción async y no devolver el token si el commit falla. No debe ocultarse `IntegrityError`/fallo de DB como éxito.

### P0-2 — Refresh rota el JTI en el JWT, pero no rota la sesión en PostgreSQL

`refresh_router.py:47-61` crea un token con un JTI nuevo, sin dependencia `AsyncSession` ni actualización de `sesiones`. El JTI anterior permanece activo y el nuevo no existe en DB. El test actual solo reemite JWT en memoria (`test_auth_refresh.py`) y no prueba el endpoint ni la transacción.

La preservación actual de `token_type=mcp` también es peligrosa: un refresh MCP produciría un JTI no persistido y rompería la validación MCP de `profile_router.py:48-75`.

**Gate:** decidir explícitamente que `/auth/refresh` solo refresca sesiones web, o implementar rotación para cada tipo. La opción recomendada para este PR es rechazar MCP en refresh. Para web: bloquear la fila antigua (`SELECT ... FOR UPDATE`), verificar que sigue activa, marcarla revocada, insertar la nueva fila y hacer un único commit. Dos refresh concurrentes del mismo JTI deben producir exactamente un éxito y un rechazo.

### P0-3 — No existe actualmente ticket WebSocket opaco ni consumo atómico de un solo uso

El router de auditoría actual solo expone REST (`/eventos`, `/eventos/{id}`, `/estadisticas`); no existe `/ws/dashboard` ni un servicio/tabla de tickets en la base actual. No es suficiente reutilizar el JWT persistente en query string o subprotocol.

El ticket debe ser un secreto aleatorio de alta entropía, entregado una sola vez, persistido únicamente como hash/HMAC y asociado a usuario, JTI de sesión, propósito, expiración y estado de consumo. El consumo debe ser atómico en PostgreSQL, idealmente mediante `UPDATE ... SET consumido_en = CURRENT_TIMESTAMP ... RETURNING ...`, con condición `consumido_en IS NULL` y expiración vigente. Dos consumidores simultáneos deben obtener como máximo una fila.

**Gate:** ticket distinto de JWT, TTL corto, sin valor crudo en DB/logs/query string, `RETURNING`, constraint/índice único y prueba de carrera. El ticket debe quedar ligado a `auditoria_sistema`, a la sesión web que lo solicitó y al usuario autenticado.

### P0-4 — El perímetro WebSocket, RBAC y Origin está sin implementar

`backend_v2/app/api/auditoria/router.py` no tiene handler WebSocket. El REST sí usa `obtener_usuario_actual_db` y comprueba `auditoria_sistema` mediante `ServicioAuth.obtener_permisos_por_rol`, pero esa política no puede asumirse automáticamente para un handshake WS.

La autenticación y el permiso efectivo deben validarse antes de `accept`; también deben revalidarse al consumir el ticket para cubrir revocación de sesión, desactivación del usuario o cambio de permisos después de emitirlo. El actor, rol y módulo nunca deben venir del cliente.

`Origin` debe validarse antes de consultar/consumir recursos, con una allowlist exacta y fail-closed en producción. La regex amplia y los orígenes hardcodeados actuales en `main.py:115-128` no son una política WS suficiente; CORS HTTP tampoco protege el handshake WS.

**Gate:** sin ticket, ticket inválido/reutilizado, usuario sin `auditoria_sistema`, usuario inactivo, sesión revocada, `Origin` ausente/no permitido o token alterado deben rechazarse antes de aceptar. No se debe reflejar el ticket como subprotocol aceptado; si se usa `Sec-WebSocket-Protocol`, se debe aceptar únicamente un protocolo constante, por ejemplo `ws-ticket`, nunca el secreto.

## 5. Hallazgos altos y riesgos de diseño

### ALTO-1 — `Sesion` almacena el JWT completo y logout busca por secreto

`Sesion.token_sesion` (`usuario.py:162-180`) almacena el token bearer completo y `marcar_fin_sesion()` lo busca por `token_sesion` (`sesion_service.py:69-87`). Esto amplía el impacto de una filtración de DB y no es apropiado para un diseño basado en JTI.

El plan debe migrar las operaciones nuevas a JTI, conservar compatibilidad controlada para filas antiguas y evitar registrar/persistir el JWT completo en nuevas sesiones. Logout debe ser idempotente y resolver el JTI desde un JWT verificado, comprobando usuario y tipo de sesión.

### ALTO-2 — `obtener_usuario_actual_db` valida persistencia solo para MCP

`profile_router.py:48-75` consulta `Sesion` únicamente cuando `token_type == "mcp"`. Los JWT web con JTI siguen siendo aceptados aunque no exista una sesión persistida o ya haya sido revocada. Para el ticket WS no debe reutilizarse esta dependencia sin una estrategia explícita: una dependencia genérica debe validar firma, `sub`, JTI, tipo, usuario activo, expiración, `fin_sesion` y correspondencia usuario/JTI.

El commit de `ultima_actividad_en` dentro de la dependencia (`:67-75`) puede confirmar cambios no relacionados y hacer rollback de la transacción del endpoint. Debe separarse de la transacción de negocio o actualizarse con una operación aislada y no bloqueante.

### ALTO-3 — Dos fuentes de verdad para JWT y configuración de Origin

`ServicioAuth` firma con `app.config.config.jwt_secret_key` (`servicio.py:182-184`), mientras `app.core.config.Settings` define `jwt_secreto`, expiración y CORS con nombres distintos. Los defaults también difieren (`app/config.py:49` frente a `app/core/config.py:33`). El plan no debe introducir un hash de ticket o validación WS basado en una tercera configuración.

Debe existir una única configuración canónica, con guard de arranque para secretos inseguros en producción, allowlist WS explícita y pruebas de alias/env. No se deben hardcodear orígenes, secretos ni valores de desarrollo.

### ALTO-4 — Migración/blindaje insuficiente y documentación desactualizada

`structural_blindaje.py:37-43` añade JTI/tipo/scope y un índice único parcial, pero `safe_execute()` ignora cualquier fallo DDL (`:7-12`). El índice actual es único para todo JTI no nulo, no solo para sesiones activas; esa semántica debe definirse antes de implementar rotación e historial.

Si PR3 agrega `tickets_ws` o columnas nuevas, se requiere DDL PostgreSQL idempotente y verificable, `TIMESTAMPTZ`, índices/constraints, rollback y validación de datos existentes. No se debe usar SQLite, `INSERT OR IGNORE` ni sintaxis de otro motor. Toda operación crítica debe aprovechar `RETURNING` cuando aplique.

`docs/ESQUEMA_BASE_DATOS.md` aún describe tablas legacy en inglés (`user_sessions`, `auth_tokens`) y no documenta la forma actual de `sesiones`, JTI ni el ticket WS. La documentación debe actualizarse junto con el modelo/migración.

### ALTO-5 — Lifecycle, cleanup y multi-worker no están resueltos

No hay servicio `auditoria` WS ni manager actual en el checkout. `main.py` solo registra startup y tareas de fondo (`:214-231`), sin shutdown/cancelación estructurada. Un manager en memoria por worker no entrega broadcasts a otros workers y puede retener sockets, tareas o conexiones Redis.

El plan debe escoger una arquitectura: Redis Pub/Sub/outbox con un manager local por worker, o retirar WS del alcance. El broadcast no puede ejecutarse dentro de `ServicioAuditoria.registrar()` ni esperar a clientes lentos. Se requieren cola acotada por cliente, timeout de envío, límite global/por usuario, desconexión de clientes lentos y cleanup en `finally` y shutdown. La sesión SQL debe vivir solo durante handshake/revalidación, nunca durante toda la conexión.

### ALTO-6 — Errores y transacciones no tienen una política única

`login_router.py:255-262` y `profile_router.py:130-136` convierten excepciones a respuestas, y el segundo expone `str(e)`. `logout` (`login_router.py:301-303`) devuelve éxito local aunque la revocación persistente falle. Para PR3, un fallo de persistencia durante emisión/consumo/rotación debe hacer rollback, no aceptar una conexión ni afirmar revocación exitosa, y devolver un error genérico sin secretos ni detalles SQL.

La auditoría de seguridad debe ejecutarse con una sesión separada o outbox, sin contaminar la transacción de auth. Nunca debe loguearse JWT, ticket, hash reversible, JTI innecesario o PII completa.

### MEDIO-1 — Scope/contratos y modularidad

Los nuevos endpoints deben recibir schemas Pydantic/SQLModel concretos; no añadir `payload: dict` ni contratos genéricos. Los nombres persistidos deben ser españoles (`hash_ticket_ws`, `consumido_en`, `expira_en`, `proposito`) salvo claims técnicos como `jti`. Mantener la cadena `router -> service -> model`; no colocar SQL, emisión, consumo o RBAC complejo en `router.py`.

`ServicioAuth` ya tiene 466 líneas. La lógica de ticket, rotación y cleanup debe extraerse a servicios especializados y mantener el límite de 550 líneas por archivo.

## 6. Flujo backend recomendado para PR3

1. **Login:** servicio async genera JTI una vez, firma JWT y registra la sesión web con el mismo JTI en la transacción del `AsyncSession` del request. El token solo se devuelve después de commit exitoso.
2. **Refresh web:** valida firma/expiración y JTI activo; bloquea la fila antigua; revoca e inserta la nueva fila en una transacción PostgreSQL; devuelve el nuevo token después del commit. MCP queda fuera de este endpoint.
3. **Logout/revocación:** opera por JTI verificado, no por JWT completo; todas las rutas de invalidación cubren sesiones web y MCP según la política acordada y son idempotentes.
4. **Emisión ticket:** `POST /api/v2/auditoria/ws-ticket`, autenticado y limitado por usuario/IP, exige `auditoria_sistema`, genera secreto opaco de un solo uso, guarda solo hash/HMAC y responde una vez con TTL corto.
5. **Handshake:** valida `Origin` exacto antes de recursos; extrae ticket sin query string; consume atómicamente; revalida JTI, usuario activo y permiso; acepta solo un subprotocol constante. Ninguna sesión DB queda abierta durante el socket.
6. **Broadcast/lifecycle:** publica una señal posterior al commit mediante outbox/Redis; el manager local usa colas acotadas y timeout; `disconnect` y shutdown cancelan tareas, cierran Redis y limpian conexiones. La pérdida de una señal no debe bloquear ni deshacer la auditoría persistida.

## 7. Tests TDD obligatorios

Todos deben vivir en `testing/backend/`, ser canónicos y registrarse en `testing/CATALOGO_PRUEBAS.md`. Al crear el servicio/controlador se exige, como mínimo, caso feliz, borde y error.

### JTI y sesiones

- Login exitoso: JTI del JWT == JTI de `sesiones`, `tipo_sesion='web'`, expiración correcta y commit único.
- Fallo de insert/commit e `IntegrityError`: rollback, no se devuelve JWT utilizable y no queda sesión huérfana.
- Login concurrente/JTI único y no colisión observable.
- Refresh web exitoso: JTI nuevo, sesión antigua revocada, nueva activa, claims permitidos preservados y una sola sesión activa.
- Dos refresh simultáneos del mismo JTI: uno `200`, otro `401`/`409`, sin duplicados.
- Refresh de JTI ausente, expirado, revocado, firma inválida y usuario inactivo: rechazo estable.
- Refresh MCP: rechazo explícito, o prueba completa de rotación persistente si el plan lo permite; nunca JTI huérfano.
- Logout por JTI, logout repetido, JTI de otro usuario y revocación masiva por cambio/reset de contraseña.
- Acceso HTTP/WS con sesión revocada, usuario desactivado o permiso revocado.

### Ticket opaco

- Emisión autorizada: secreto no predecible, no se persiste crudo, TTL y propósito correctos, usuario/JTI ligados y respuesta concreta.
- Sin token, sin `auditoria_sistema`, usuario inactivo, sesión no persistida o rate limit excedido: `401`/`403`/`429` según contrato.
- Ticket vencido, malformado, de otro propósito, de otra sesión y ya consumido: rechazo sin filtrar el valor.
- Consumo concurrente del mismo ticket: exactamente un éxito; el otro no puede aceptar el WS.
- Falla DB durante emisión/consumo: rollback y rechazo; el ticket no queda parcialmente utilizable.
- Revocar sesión o permiso después de emitir y antes de consumir: consumo rechazado.
- El ticket no aparece en logs, auditoría, query string ni subprotocol aceptado.

### WebSocket, Origin y lifecycle

- `Origin` exacto permitido, ausente, `null`, puerto no permitido, origen parecido/malicioso y allowlist de producción.
- Handshake sin ticket, ticket inválido/revocado/reutilizado y usuario sin permiso: rechazo antes de `accept`.
- Conexión autorizada: subprotocol constante, nunca el ticket; no mantiene `AsyncSession` abierta.
- Broadcast con cliente lento: timeout/desconexión o coalescing, sin bloquear `ServicioAuditoria.registrar()`.
- Límite global/por usuario, mensajes sobredimensionados, desconexión repetida y cleanup en `finally`.
- Revalidación periódica de sesión/RBAC y cierre al revocar; shutdown cancela tareas, cierra pub/sub y deja cero conexiones registradas.
- Comunicación entre workers o prueba que documente explícitamente la decisión de no soportar multi-worker.

### Auditoría, RBAC y regresión

- HTTP real `401/403/200` para `/eventos`, `/eventos/{id}`, `/estadisticas` y `/ws-ticket`; no solo mocks de la dependencia.
- Actor siempre derivado del principal; body con `usuario_id`, `rol` o `nombre` falsos no altera el evento.
- Enmascarado de JWT, ticket, JTI sensible, cédula y secretos en DB, respuesta y logs.
- Fallo de auditoría no rompe la operación definida; rollback deja la sesión reutilizable.
- PostgreSQL real para constraints, `TIMESTAMPTZ`, índices, `FOR UPDATE`, `UPDATE ... RETURNING` y concurrencia; no SQLite.
- `test_infrastructure.py`, `test_regresiones.py`, `test_auth_rate_limit.py`, `test_auth_refresh.py` y suites focales de auditoría.
- Colección sin duplicados y DB exclusivamente de pruebas; ningún test debe caer a `.env` real.

La colección focalizada actual solo encontró 15 tests de refresh, verificación y estadísticas HTTP; no existe una suite WS/JTI transaccional en la base actual. La colección se realizó, pero no se ejecutaron tests, Docker ni Alembic en esta revisión read-only.

## 10. Blocking reasons

1. El login actual no enlaza JTI del JWT con la fila de sesión y el servicio puede ocultar el fallo de persistencia.
2. Refresh crea JTI nuevos sin rotación persistente y permite una semántica MCP inconsistente.
3. No existe el ticket opaco de un solo uso ni una tabla/operación atómica de consumo.
4. El canal WebSocket de auditoría, su RBAC, Origin, límites, multi-worker y cleanup no existen en el backend actual.

**Resultado final: bloqueado.**
